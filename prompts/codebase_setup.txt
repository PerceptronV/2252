     Graph Partitioning Research Skeleton

     Context

     You are starting a CS 2252 graph partitioning study (NeurIPS 2026
     submission per docs/main.tex) and want to compare many algorithms across
     many datasets and metrics. The repo currently contains a single ad-hoc
     utility (core/spectral_embedding.py) and no scaffolding for structured
     experiments.

     This plan creates a modular, plug-and-play skeleton along three axes —
     dataset, algorithm, eval — driven by YAML configs as the single source of
     truth, with results written to UUID-stamped run directories so future
     comparisons are tractable. Today only the base classes and the runner are
     wired up; concrete datasets (SBM, LFR, Karate, Cora), concrete algorithms
     (spectral, Cheeger sweep cut), and concrete evals (conductance, ARI) land
     later by subclassing and registering.

     Design decisions confirmed via AskUserQuestion: Graph dataclass;
     pyproject.toml + editable install; move existing core/spectral_embedding.py
      into src/algorithms/_spectral.py; list of evals per run.

     Core Abstractions

     src/core/graph.py — Graph dataclass

     @dataclass(frozen=True)
     class Graph:
         adjacency: sp.csr_matrix          # shape (n, n), symmetric, zero 
     diagonal
         num_nodes: int
         node_features: np.ndarray | None = None   # shape (n, d), optional 
     (Cora etc.)
         name: str | None = None                   # human-readable label

     Frozen so it's safe to pass between dataset → algorithm → eval. A
     __post_init__ validates that adjacency is square and matches num_nodes.

     src/data/base.py — Dataset ABC

     class Dataset(ABC):
         @abstractmethod
         def load(self) -> tuple[Graph, np.ndarray]:
             """Return (graph, target_partition). target_partition has shape 
     (n,) of int labels."""
         @property
         @abstractmethod
         def num_clusters(self) -> int: ...
         name: str   # set by @register_dataset

     Subclasses take their hyperparameters via __init__(**params) so they can be
      constructed from config.

     src/algorithms/base.py — Algorithm ABC

     class Algorithm(ABC):
         @abstractmethod
         def fit_predict(self, graph: Graph, k: int) -> np.ndarray:
             """Return predicted labels of shape (n,)."""
         name: str

     src/evals/base.py — Eval ABC

     class Eval(ABC):
         @abstractmethod
         def __call__(
             self,
             graph: Graph,
             predicted: np.ndarray,
             target: np.ndarray,
         ) -> float: ...
         name: str

     Single uniform signature; conductance ignores target, ARI ignores graph.
     Keeps the runner's call site identical for every eval.

     src/core/registry.py — Registries

     Three module-level dicts (DATASETS, ALGORITHMS, EVALS) plus three
     decorators:

     @register_dataset("sbm")
     class SBMDataset(Dataset): ...

     The decorator stores the class under its registry name and sets cls.name = 
     "sbm". The runner resolves config strings via DATASETS["sbm"](**params),
     etc.

     Registration is triggered by importing the package. To avoid each subclass
     needing to be imported explicitly, src/data/__init__.py,
     src/algorithms/__init__.py, and src/evals/__init__.py will import every
     concrete module in their package (just _spectral for now; later they'll
     import sbm.py, conductance.py, etc.). That single import-side-effect
     pattern keeps configs decoupled from import order.

     Config Format (configs/example.yaml)

     name: sbm_spectral_baseline           # human label, copied into results
     seed: 42

     dataset:
       type: sbm                            # registry key
       params:                              # forwarded as **kwargs to the class
         n: 200
         k: 4
         p_in: 0.3
         p_out: 0.05

     algorithm:
       type: spectral
       params:
         embedding_dim: 4
         row_normalize: true

     evals:
       - type: conductance
       - type: ari

     src/core/config.py parses this into a typed ExperimentConfig dataclass. No
     concrete dataset/algorithm/eval is implemented in this PR, so the example
     config references types that don't yet exist — it's a template, not a
     runnable example. A second config, configs/_smoke.yaml, references three
     trivial built-in stubs (a "trivial" dataset that returns a 4-node graph, an
      "all_zeros" algorithm, and an "accuracy" eval) so the runner has something
      to exercise end-to-end. The smoke stubs live alongside the base classes
     (e.g. src/data/base.py adds a _TrivialDataset) and are explicitly marked as
      test-only.

     Experiment Runner (src/core/runner.py)

     Pseudocode:

     def run_experiment(config_path: Path) -> Path:
         cfg = load_config(config_path)
         rng_seed = cfg.seed

         run_uuid = uuid.uuid4().hex
         run_dir = Path("results") / run_uuid
         run_dir.mkdir(parents=True)

         dataset   = DATASETS[cfg.dataset.type](**cfg.dataset.params)
         algorithm = ALGORITHMS[cfg.algorithm.type](**cfg.algorithm.params)
         evals     = [EVALS[e.type](**e.params) for e in cfg.evals]

         graph, target = dataset.load()
         predicted = algorithm.fit_predict(graph, k=dataset.num_clusters)

         scores = {ev.name: float(ev(graph, predicted, target)) for ev in evals}

         write_results(run_dir, cfg, graph, predicted, target, scores)
         return run_dir
